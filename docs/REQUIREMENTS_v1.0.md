## كيفية قراءة الوثيقة
تستخدم هذه الوثيقة معرفات ثابتة للمتطلبات. يبدأ كل متطلب وظيفي بالرمز FR، وكل قاعدة عمل بالرمز BR، وكل متطلب غير وظيفي بالرمز NFR. الأولوية Must تعني إلزامي للإصدار الإنتاجي، وShould تعني مهم ويمكن تنفيذه بعد النواة، وCould تعني تحسين مستقبلي.
تم الفصل بين «الحالي المؤكد من الكود» و«المستهدف النهائي». الهدف ليس نسخ Client Script كما هو، بل تحويل الفكرة الحالية إلى نظام إنتاجي آمن، قابل للتدقيق والتوسعة، مع الحفاظ على السلوك المتفق عليه.
## 1. الرؤية والهدف
بناء نظام متكامل لإدارة دورة طلبات قص ألواح MDF/الخشب من لحظة إدخال طلب العميل وحتى إتمام القص والقشاط والمراحل الإنتاجية، مع تحسين توزيع القطع على الألواح لتقليل عدد الألواح والهدر، احتساب التكلفة بدقة، إدارة المخزون والبقايا، تتبع العامل والزمن والأخطاء والتعويضات، وإصدار مخرجات تشغيلية دقيقة للطباعة وملفات DXF.
- تقليل الهدر وعدد الألواح المستخدمة قدر الإمكان مع احترام قيود القص الفعلية.
- منع الأخطاء الناتجة عن الإدخال اليدوي أو اختلاف القياسات والوحدات.
- إعطاء العامل خطة قص واضحة وقابلة للتنفيذ، مع ترقيم موحد للقطع.
- ربط العملية بالمخزون والتكاليف والإنتاج بدل بقاء الحساب في Client Script فقط.
- إتاحة أثر تدقيقي كامل: من أنشأ، من راجع، من قص، من قشّط، متى بدأ وانتهى، وما الذي تغيّر.
- دعم العربية والإنجليزية في واجهة النظام والمصطلحات الأساسية.
## 2. نطاق النظام
### 2.1 داخل النطاق
- إدارة طلبات القص وربطها بالعميل والتاريخ والحالة.
- إدارة صنف اللوح: المادة، اللون، السماكة، الأبعاد، السعر، المخزون، وإشارة Is MDF Board.
- إدخال الدرف/القطع بالسنتيمتر مع الكمية، القشاط، التدوير والملاحظات.
- الحسابات التلقائية للمساحة، أمتار القشاط، تكلفة القشاط، عدد الألواح، الهدر، تكلفة اللوح والقص والإجمالي.
- محرك 2D Rectangle Packing/Nesting متعدد الخوارزميات مع Auto لاختيار أفضل نتيجة.
- رسم خطة القص ولوحات متعددة مع إظهار القياسات، الترقيم والقشاط.
- طباعة خطة القص وجدول القياسات.
- تصدير DXF للاستخدام في CAD/CNC.
- حجز/خصم المخزون وفق حالة العمل الفعلية، وليس بمجرد فتح النموذج.
- إدارة بقايا الألواح Board Remnants وإعادة استخدامها حسب المادة/اللون/السماكة.
- إدارة مراحل الإنتاج وحالات Start/Pause/Resume/Finish والعامل والزمن.
- معالجة الأخطاء والقطع المفقودة والتعويضات الداخلية المرتبطة بالطلب الأصلي.
- التقارير ولوحات المتابعة والصلاحيات وسجل التدقيق.
### 2.2 خارج النواة الحالية / تطوير لاحق
- التحكم المباشر بآلة CNC عبر بروتوكول خاص بالآلة؛ النواة تصدر DXF صحيحًا، ويضاف التكامل المباشر عند تحديد نوع الماكينة والبروتوكول.
- تحسين ثلاثي الأبعاد أو Nesting لأشكال غير مستطيلة؛ النطاق الحالي مستطيلات.
- التجارة الإلكترونية أو بوابة العميل العامة؛ ليست جزءًا من النواة إلا إذا أضيفت لاحقًا.

## 5. الافتراضات والقيود العامة
- النظام يعمل كتطبيق مخصص داخل Frappe / ERPNext، مع الاستفادة من Customer وItem وWarehouse وStock Entry والصلاحيات القياسية حيث يكون ذلك مناسبًا.
- أبعاد القطع مستطيلة في الإصدار 1.0.
- الإدخال الأساسي للقطعة بالسنتيمتر، وأبعاد اللوح المرجعية بالملليمتر، ويجب منع الخلط الصامت بين الوحدات.
- الاتجاه الافتراضي للقطعة طولي: عرض القطعة على محور عرض اللوح X، وطول القطعة على محور طول اللوح Y.
- لا يسمح بالتدوير إلا عند تفعيل Allow Rotation للقطعة/المجموعة.
- يجب احترام Kerf وTrim Margin في جميع الحسابات ومخرجات DXF وخطة القص.
- أي عملية مخزنية أو مالية فعلية يجب أن تتم Server-side وبصلاحية موثوقة، وليس اعتمادًا على Client Script فقط.
- يجب حفظ Snapshot لخطة القص المعتمدة حتى لا تتغير نتيجة الطلب التاريخي عند تعديل الخوارزميات مستقبلًا.

## 9. الحسابات الأساسية
### 9.1 مساحة القطع
مساحة المجموعة بالمتر المربع = Width CM × Length CM × Quantity ÷ 10000.
Total Area M2 = مجموع Area M2 لجميع المجموعات الصحيحة.

### 9.2 أمتار القشاط
عدد حواف الطول = Edge Long Right + Edge Long Left. عدد حواف العرض = Edge Width Top + Edge Width Bottom.
Edge Meters = ((Length CM × عدد حواف الطول) + (Width CM × عدد حواف العرض)) × Quantity ÷ 100.

### 9.3 الدقة والتقريب
القيم الداخلية لا تُقرب قبل انتهاء المعادلة المركبة. العرض الحالي يفضل 3 منازل للمساحة/الأمتار/التكلفة و2 لنسبة الهدر، مع استخدام دقة ERPNext المالية عند الترحيل المحاسبي.

### 13.2 وضع Auto

## 19. التكلفة
Baseline:
MDF Cost = New Full Boards Used × Board Rate، مع سياسة مستقلة لتكلفة Remnants.
Cutting Cost = عدد الألواح/المصادر الخاضعة للرسم × Cutting Cost Per Board حسب إعداد التسعير.
Edge Cost = مجموع Edge Meters × Effective Edge Rate لكل مجموعة/قطعة.
Total Cost = MDF Cost + Cutting Cost + Edge Cost + تكاليف إضافية معتمدة إن كانت ضمن إعدادات الإصدار.

## 31. الحالات الحدية الواجب معالجتها
- قطعة أكبر من usable board في الاتجاهين → Unplaceable ورسالة تحدد القطعة.
- قطعة تدخل فقط بعد Rotation وAllow Rotation=false → مرفوضة مع اقتراح تفعيل التدوير إن كان مناسبًا.
- Trim يساوي/يتجاوز نصف البعد → أبعاد قابلة للاستخدام غير صالحة.
- Kerf سالب أو غير منطقي → رفض الإدخال.
- Quantity كسري → رفض.
- Edge Type محذوف/غير نشط بعد إنشاء Draft → يمنع الاعتماد أو يستخدم Snapshot إذا كان الطلب معتمدًا.
- تغيير سعر القشاط بعد اعتماد الطلب → لا يغير التاريخ.
- تغيير أبعاد Board Item بعد اعتماد خطة → لا يغير Snapshot.
- ضغط Recalculate عدة مرات بسرعة → النتيجة الأحدث فقط هي المعروضة.
- طلبان يحاولان نفس Remnant → واحد فقط ينجح بالحجز.
- إلغاء طلب بعد استهلاك مخزون → مسار عكس مضبوط، لا حذف مباشر.
- DXF مع عدة ألواح → لا تداخل في الإحداثيات/الملفات وتحديد واضح لكل Sheet.
- ملاحظات تحتوي HTML/Script → تعرض كنص آمن.
- قطع متساوية جدًا مع floating precision → استخدام tolerance موحد في المقارنات الهندسية.

## 32. معايير القبول الحسابية – أمثلة مرجعية
### AC-01 مساحة مجموعة
المدخل: 50×90 سم، كمية 3. المتوقع: Area = 50×90×3÷10000 = 1.35 م².
### AC-02 أمتار القشاط
المدخل: 50×90، كمية 1، قشاط يمين+يسار+أعلى. المتوقع: (90+90+50)÷100 = 2.3 م.
### AC-03 القشاط الافتراضي والمخصص
Header = 2سم عادي 0.5$/m. Row A بلا نوع → 0.5. Row B = 4سم ذهبي 2.5$/m → يستخدم 2.5 فقط لـB. Total Edge Cost = مجموع تكلفة الصفين.
### AC-04 الترقيم
السطر الأول qty=3 → 1.1,1.2,1.3. السطر الثاني qty=2 → 2.1,2.2. حذف/إعادة ترتيب السطور قبل الاعتماد يعيد التوليد بصورة واضحة؛ بعد الاعتماد Snapshot ثابت.
### AC-05 Trim
لوح 2800×2070 mm وTrim=5mm لكل جهة → usable = 2790×2060 mm.
### AC-06 Rotation
قطعة لا تدخل 80×120 في مساحة 100×90، لكنها تدخل 120×80 فقط إذا كانت المساحة/الاتجاه يسمحان وبعد تفعيل Allow Rotation. بدون التفعيل تبقى Unplaced.
### AC-07 Auto
إذا طريقة A تعطي 2 ألواح وهدر أقل، وطريقة B تعطي 1 لوح دون Unplaced، Auto يختار B لأن عدد الألواح أعلى أولوية من الهدر. أي طريقة فيها Unplaced تخسر أمام طريقة بلا Unplaced.
### AC-08 الاستقلال عن HTML
حذف cutting_plan_html cache وإعادة فتح الطلب المعتمد يجب أن يسمح بإعادة توليد نفس الرسم من Plan Snapshot.
### AC-09 المخزون
Recalculate عشر مرات لا يغير Stock Ledger. فقط الحدث المعتمد حسب Policy ينشئ Reservation/Consumption مرة واحدة.
### AC-10 التعويض
قطعة 2.3 تالفة بسبب خطأ قص. ينشأ Incident + Replacement مرتبطان بـ2.3، يبحث النظام عن Remnant مطابق، وتظهر التكلفة Internal Loss ولا تضاف إلى Customer Charge.

هذه ليست أسئلة تعيق بدء التطوير؛ وضعت Defaults آمنة، لكن يجب تسجيل القرار النهائي قبل الإنتاج:
1. نقطة خصم اللوح فعليًا: عند Start Cutting (مقترح) أم Approval؟
2. هل الحجز Reservation يتم عند Approval؟ المقترح نعم.
3. سياسة تقييم Remnants محاسبيًا ومن يتحمل تكلفة اللوح الكامل.
4. الحد الأدنى لحفظ Remnant حسب الطول/العرض/المساحة.
5. هل Cutting Cost يحسب لكل Full Board فقط أم لكل Source Sheet بما فيها Remnant؟
6. مراحل الإنتاج الإضافية الإلزامية بعد القشاط حسب واقع المعمل.
7. نوع ماكينة CNC وبرنامجها وصيغة DXF المقبولة فعليًا.
8. Warehouses/Locations الفعلية للألواح والقشاط والبقايا.

## 39. تعريف Done للإصدار الأول الإنتاجي
لا يعتبر النظام جاهزًا للإنتاج لمجرد أن الرسم يظهر. يعتبر Done عندما:
- جميع متطلبات Must الأساسية منفذة ومختبرة أو يوجد Waiver رسمي.
- لا توجد خطة معتمدة فيها overlap/out-of-bounds/unplaced.
- المخزون لا يتغير من Preview/Recalculate ولا توجد خصومات مكررة.
- الخطة المعتمدة قابلة لإعادة الطباعة والتصدير من Snapshot ثابت.
- القشاط والتكلفة يطابقان عينات حساب يدوية معتمدة.
- Remnants لا تستخدم مرتين ومتوافقة مع المادة/اللون/السماكة.
- دورة Start/Pause/Resume/Finish والعامل والزمن قابلة للتدقيق.
- Replacement لقطعة تالفة يعمل من الحدث حتى الإكمال ولا يحمّل العميل خطأ المعمل.
- الصلاحيات تمنع API غير المصرح حتى لو حاول المستخدم تجاوز الواجهة.
- Backup/Restore وUAT على بيانات شبه حقيقية ناجحان.
- DXF مفتوح ومقاساته صحيحة على البرنامج/الماكينة المستهدفة.

## 40. الخلاصة التنفيذية
الكود الحالي يمثل نواة جيدة جدًا لمشكلة Door Cutting Optimization: حساب الدرف والقشاط، 17 طريقة Packing تقريبًا، Auto comparator، رسم، طباعة وDXF. لكنه يجمع مسؤوليات كثيرة في Client Script ويحسب الهدر بصورة مبسطة ولا يشكل وحده نظام تصنيع كامل.
المنتج المستهدف يحافظ على نفس تجربة الإدخال والسلوك المفيد، لكنه ينقل مصدر الحقيقة إلى بنية موثوقة: Structured Cutting Plan + Server-side validation + Stock/Remnants + Production Stages + Replacement + Audit. بهذه البنية يصبح النظام عمليًا للمعمل، قابلًا للتوسع والاختبار، ولا يعتمد نجاحه على صفحة متصفح واحدة.
